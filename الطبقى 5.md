يا يونس، ها هي **الطبقة الخامسة: محرك التوافق (Compatibility Subsystem / EXE Engine)**، وهي **"المترجم الذكي"** اللي هيخلي مدار يشغل ملفات الويندوز من غير ما يهد النواة.

الخطة مصممة بنفس الدقة المعمارية، ومدمج فيها **بروتوكولات العزل والأمان** عشان نضمن استقرار النظام حتى لو كان ملف الـ `exe` تالف أو خبيث.

---

## 🔄 **الطبقة الخامسة: محرك التوافق (Compatibility Subsystem)** (النسخة المفصّلة)

### 🎯 **الهدف العام**
تحويل ملفات `.exe` (Windows PE Format) إلى تطبيقات قابلة للتنفيذ على مدار، عبر:
- تحليل بنية ملفات PE وتحميلها في **فضاء المستخدم (User-Space)**
- إنشاء طبقة ترجمة (API Translation Stubs) تحول استدعاءات Win32 الأساسية إلى `syscalls` خاصة بمدار
- ربط استثناءات ويندوز (SEH) بنظام معالجة الأخطاء في مدار
- دمج خيوط التطبيق مع جدول المهام (Scheduler)
- اختبار الاستقرار مع ملفات `.exe` حقيقية بسيطة

### 🚧 **الحدود الصارمة (Boundaries)**
| ✅ مسموح في هذه الطبقة | ❌ ممنوع تماماً |
|------------------------|----------------|
| تشغيل كـ `compatd` Daemon في User-Space | تنفيذ أي كود EXE داخل النواة (Kernel Panic خطر) |
| ترجمة ~20 دالة Win32 أساسية فقط في المرحلة الأولى | دعم كامل لـ DLLs مع Relocations أو COM/ActiveX |
| عزل كل عملية EXE بـ Sandbox افتراضي | تعديل ذاكرة النواة أو تعريفات الهاردوير مباشرة |
| تحويل الأخطاء لـ `ExitProcess` نظيف | ترك الاستثناءات تهرب للنواة بدون معالجة |

## 🧩 **المهام الدقيقة (6 مهام)**

### 🔹 المهمة 1: محلل ملفات PE (Parser & Validator)
#### 📁 الملفات المطلوبة
```
compat/
├── pe_parser.c / pe_parser.h
├── pe_structs.h              (تعريفات DOS, NT, Optional, Section)
└── pe_validator.c            (فحص التوقيع، المعمارية، والسلامة)
```
#### 🔧 المواصفات التقنية
- **القراءة:** `DOS Header` → `PE Signature ("PE\0\0")` → `COFF Header` → `Optional Header (PE32+ 0x20B)` → `Section Table`
- **الاستخراج:** `ImageBase`, `AddressOfEntryPoint`, `Section RVA/Size`, `Characteristics`
- **التحقق:** رفض الملفات 32-bit مؤقتاً، فحص حدود الـ `RVA` عشان متخرجش عن حجم الملف

#### ✅ معيار النجاح
```
🧪 Testing PE Parser...
  File: test64.exe → Valid PE32+
  Sections: .text(RX), .data(RW), .rdata(R)
  Entry Point: 0x140001000
✅ PE Parser PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `RVA` خارج حدود الملف | أضف `if (rva > file_size) return ERROR_INVALID_IMAGE` |
| توقيع PE مفقود | اطبع `Error: Not a valid PE file` وأوقف التحميل فوراً |

---

### 🔹 المهمة 2: تحميل الأقسام وتعيين الذاكرة (Section Mapping)
#### 📁 الملفات المطلوبة
```
compat/
├── pe_loader.c / pe_loader.h
├── mem_mapper.c              (ربط مع sys_mmap من الطبقة 2)
└── stack_init.c              (تهيئة Kernel/User Stack للعملية)
```
#### 🔧 المواصفات التقنية
- **التخصيص:** استدعاء `sys_mmap` لكل قسم بصلاحيات مطابقة (`.text` → `PROT_READ|PROT_EXEC`, `.data` → `PROT_READ|PROT_WRITE`)
- **الـ BSS:** تخصيص ذاكرة صفرية بنفس حجم القسم
- **التهيئة:** ضبط `RIP` على `EntryPoint`، `RSP` على Stack جديد، وتحميل الـ `Environment Block`

#### ✅ معيار النجاح
```
🧪 Testing Section Mapping...
  Mapped .text at 0x400000 (RX)
  Mapped .data at 0x401000 (RW)
  Entry Point reachable → No Page Fault
✅ Memory Mapping PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `Page Fault` عند التنفيذ | راجع صلاحيات الصفحات (`PTE flags`) وتأكد إن الـ `NX bit` معطل للأقسام التنفيذية |
| تعارض عناوين مع النظام | فعّل `ASLR` بسيط أو ابدأ التحميل من `0x400000` ثابت مؤقتاً |

### 🔹 المهمة 3: ترجمة الـ API الأساسية (Win32 → Orbit Syscalls)
#### 📁 الملفات المطلوبة
```
compat/
├── win32_stubs.c / win32_stubs.h
├── syscall_bridge.c          (جدول التحويل الموحد)
└── api_log.c                 (تسجيل الاستدعاءات للتصحيح)
```
#### 🔧 المواصفات التقنية
- **الجدول الأولي (20 دالة):**
  `CreateFileA`, `ReadFile`, `WriteFile`, `CloseHandle`, `ExitProcess`, `GetCurrentProcess`, `VirtualAlloc`, `VirtualFree`, `MessageBoxA`, `GetStdHandle`, `WriteConsoleA`, `Sleep`, `GetTickCount`, `CreateThread`, `ExitThread`, `WaitForSingleObject`, `GetModuleHandleA`, `HeapAlloc`, `HeapFree`, `SetLastError`
- **آلية الجسر:** كل Stub ياخد باراميترات Win32 → يحوّلها لـ `struct syscall_args` → يستدعي `sys_call` → يرجع النتيجة بنفس صيغة Win32 (`BOOL`, `HANDLE`, `DWORD`)

#### ✅ معيار النجاح
```
🧪 Testing API Bridge...
  CreateFileA("test.txt") → HANDLE=0x1
  WriteFile(0x1, "Madar", 5) → OK
  MessageBoxA("Hello") → Window shown (via Layer 6 syscall)
  ExitProcess(0) → Clean exit, code=0
✅ API Bridge PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| دالة غير مدعومة | ارجع `ERROR_CALL_NOT_IMPLEMENTED` وسجلها في `api_log` للتطوير لاحقاً |
| تحويل الباراميترات خاطئ | استخدم `#pragma pack(push, 1)` لهياكل Win32 عشان الـ Alignment متلخبطش |

### 🔹 المهمة 4: معالجة الاستثناءات (SEH Bridge)
#### 📁 الملفات المطلوبة
```
compat/
├── seh_bridge.c / seh_bridge.h
├── exception_mapper.c        (ربط IDT Exceptions بـ Win32 Codes)
└── safe_exit.c               (إنهاء نظيف عند الكراش)
```
#### 🔧 المواصفات التقنية
- **الربط:** معالج الاستثناءات في النواة (`#DE`, `#PF`, `#GP`) يرسل إشارة لـ `compatd`
- **المحاكاة:** استخدام `setjmp/longjmp` أو Signal Handlers لمحاكاة `__try/__except`
- **العزل:** أي استثناء غير معالج → يقتل عملية الـ EXE فقط → يرجع `EXCEPTION_ACCESS_VIOLATION` → النواة مستمرة

#### ✅ معيار النجاح
```
🧪 Testing Exception Handling...
  Trigger: Divide by Zero inside EXE
  Caught by SEH Bridge → Handler executed
  Process exited cleanly → Kernel stable
✅ SEH Bridge PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| الاستثناء يهد النواة | تأكد إن الـ Handler بيستخدم `iretq` صحيح وبيستعيد الـ `RSP` القديم |
| تسرب موارد بعد الكراش | فعّل `cleanup_on_crash()` في `compatd` يحرر الذاكرة ويقفل الـ Handles |

### 🔹 المهمة 5: دمج الخيوط والـ TLS (Threading & Local Storage)
#### 📁 الملفات المطلوبة
```
compat/
├── pe_threads.c / pe_threads.h
├── tls_manager.c             (محاكاة TEB/PEB بسيط)
└── sync_mapper.c             (ربط Win32 Sync Objects بـ Layer 2)
```
#### 🔧 المواصفات التقنية
- **الخيوط:** `CreateThread` → `sys_thread_create(entry, stack, args)`
- **الـ TLS:** جدول `thread_local_vars` لكل خيط، يتم الوصول له عبر `fs:[0x30]` (محاكاة PEB)
- **المزامنة:** `WaitForSingleObject` → يترجم لـ `sys_wait_thread` أو `sys_sleep`

#### ✅ معيار النجاح
```
🧪 Testing Threading & TLS...
  Created 2 threads → Both running
  TLS var: Thread A=10, Thread B=20 (isolated)
  WaitForSingleObject → Returned after thread exit
✅ Threading & TLS PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| تعارض في الـ Scheduler | حدّد أولوية خيوط الـ EXE أقل من خيوط النواة (`priority = USER_NORMAL`) |
| TLS يتلخبط بين الخيوط | استخدم `thread_id` كمفتاح في جدول الـ TLS، مش مؤشر ثابت |

### 🔹 المهمة 6: التكامل واختبار `.exe` حقيقي (Integration & Sandbox)

## 🛡️ **البروتوكول الوقائي المدمج في الطبقة 5**

1. ✅ **User-Space Only:** `compatd` وكل مكتبات الترجمة تعمل بعيداً عن النواة. كراش الـ EXE = موت الـ EXE فقط.
2. ✅ **Minimal API Set:** نبدأ بـ 20 دالة تغطي 80% من البرامج البسيطة. التوسع تدريجي بناءً على `api_log`.
3. ✅ **Strict Syscall Boundary:** لا اتصال مباشر مع Layer 2/4 إلا عبر `sys_call` موحد ومعروف.
4. ✅ **Sandbox Limits:** ذاكرة، CPU، وملفات محددة. أي تجاوز → قتل فوري.
5. ✅ **Logging First:** كل استدعاء Win32 يُسجل (`api_log.c`) عشان نعرف إيه الناقص ونطوّره بدقة.
