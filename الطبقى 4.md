يا يونس، ها هي **الطبقة الرابعة: خدمات النظام (System Services)**، وهي **"البنية التحتية البرمجية"** اللي هتستقبل عليها كل التطبيقات لاحقاً. هنا بيبدأ النظام يتحول من "مجموعة تعريفات" إلى **منصة خدمات متكاملة**.

الخطة مصممة بنفس الدقة، ومدمج فيها **بروتوكولات الحماية** عشان متغرقش في تعقيدات الخدمات.

---

## 🏗️ **الطبقة الرابعة: خدمات النظام (System Services)** (النسخة المفصّلة)

### 🎯 **الهدف العام**
تحويل النواة من مُدير موارد خام إلى **منصة خدمات منظمة** تدعم:
- نظام ملفات موحد (VFS) يعمل كواجهة لكل أنظمة التخزين
- دعم فعلي لنظام ملفات قياسي (FAT32 Minimal) للقراءة والكتابة
- قنوات اتصال آمنة بين العمليات (IPC) وإدارة الإشارات
- مكدس شبكة أساسي (ARP + IPv4 + ICMP) للاتصال الخارجي
- نموذج صلاحيات بسيط (UID/GID) لحماية الموارد
- عملية التهيئة (`init`) وإدارة الخدمات الخلفية تلقائياً

### 🚧 **الحدود الصارمة (Boundaries)**
| ✅ مسموح في هذه الطبقة | ❌ ممنوع تماماً |
|------------------------|----------------|
| VFS abstraction + FAT32 parser | دعم NFS, SMB, أو NTFS المعقد |
| Pipes, Message Queues, Signals | تشفير TLS/SSL أو بروتوكولات متقدمة |
| ARP, IPv4, ICMP Echo/Reply | TCP/UDP الكامل (يؤجل للطبقة 7) |
| UID/GID table + permission bits | دمج واجهة المستخدم أو تطبيقات هنا |
| Init process (PID 1) logic | أي اعتماد على `glibc` أو `systemd` |

## 🧩 **المهام الدقيقة (6 مهام)**

### 🔹 المهمة 1: تجريد نظام الملفات (VFS Core)
#### 📁 الملفات المطلوبة
```
services/
├── vfs.h / vfs.c          (الواجهة الموحدة)
├── inode.h / dentry.h     (هياكل البيانات)
└── path_resolver.c        (تحليل المسارات /dir/file)
```
#### 🔧 المواصفات التقنية
- **الواجهة:** `vfs_open()`, `vfs_read()`, `vfs_write()`, `vfs_close()`, `vfs_mount()`
- **التخزين المؤقت:** RAM Disk بسيط للاختبار قبل ربطه بـ FAT32
- **تحليل المسار:** تحويل `/home/test.txt` إلى Inode صحيح مع دعم `.` و `..`

#### ✅ معيار النجاح
```
🧪 Testing VFS Core...
  open("/test.txt") → OK
  write("Madar VFS") → 10 bytes
  read() → "Madar VFS" (exact match)
✅ VFS Core PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `vfs_open` يفشل مع مسارات متداخلة | راجع دالة `path_normalize()` وتأكد من معالجة الـ `/` المتكررة |
| تسرب Inodes | أضف عداد `inode_refcount` وتأكد من `vfs_close()` ينقصه |

---

### 🔹 المهمة 2: دعم نظام ملفات فعلي (FAT32 Minimal)
#### 📁 الملفات المطلوبة
```
services/
├── fat32.h / fat32.c      (قراءة BPB، FAT Tables، Data Clusters)
├── disk_io.h              (واجهة للقراءة من Layer 3 ATA)
└── fat_dir.c              (قراءة المجلدات وملفات الجذر)
```
#### 🔧 المواصفات التقنية
- **القراءة:** تحليل `BIOS Parameter Block`، تحديد موقع `FAT1/2`، `Root Directory`، و `Data Region`
- **السلاسل:** تتبع Cluster Chain للوصول للملفات المتفرقة
- **الكتابة:** تحديث FAT Table + Data Clusters + Directory Entry

#### ✅ معيار النجاح
```
🧪 Testing FAT32 Driver...
  Mounting fat32.img...
  Listing Root: [HELLO.TXT] [BOOT.BIN]
  Reading HELLO.TXT → "Hello from Madar!"
✅ FAT32 Driver PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| الملفات تظهر تالفة | راجع حساب `ClusterSize = SectorsPerCluster * BytesPerSector` |
| الكتابة تفسد الجدول | فعّل `FAT Mirror` مؤقتاً (اكتب في FAT1 و FAT2 معاً) |

---

### 🔹 المهمة 3: الاتصال بين العمليات (IPC) + الإشارات
#### 📁 الملفات المطلوبة
```
services/
├── ipc.h / ipc.c          (Pipes, Message Queues)
├── signal.h / signal.c    (SIGTERM, SIGKILL, handlers)
└── ring_buffer.h          (ذاكرة مؤقتة آمنة)
```
#### 🔧 المواصفات التقنية
- **Pipes:** `pipe_create()` → يعيد قناتي قراءة/كتابة. `pipe_write()` و `pipe_read()` غير متزامنة.
- **الإشارات:** جدول `signal_handlers` لكل عملية. `sys_kill(pid, sig)` يرسل إشارة.
- **الأمان:** Spinlock على كل Buffer لمنع Data Race.

#### ✅ معيار النجاح
```
🧪 Testing IPC & Signals...
  Pipe: Thread A sends 100 msgs → Thread B receives 100 (0 loss)
  Signal: SIGTERM handled gracefully → Process exits cleanly
✅ IPC & Signals PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| Deadlock في الـ Pipe | أضف `timeout_ms` لـ `pipe_read()`، وأطلق الـ Lock قبل النوم |
| إشارة تضيع | استخدم `pending_signal_mask` bitfield بدل قائمة بسيطة |

---

### 🔹 المهمة 4: المكدس الشبكي الأساسي (ARP + IPv4 + ICMP)
#### 📁 الملفات المطلوبة
```
services/
├── net_stack.h            (هياكل Ethernet/IP/ICMP)
├── arp.c / arp.h          (Cache, Request, Reply)
├── ip.c / ip.h            (Header parsing, Checksum, Routing بسيط)
└── icmp.c / icmp.h        (Echo Request/Reply)
```
#### 🔧 المواصفات التقنية
- **ARP:** جدول `MAC-IP`، إرسال `Who has IP?`، استقبال `Reply` وتحديث الكاش
- **IPv4:** تحليل الهيدر، التحقق من `Header Checksum`، توجيه بسيط (Default Gateway)
- **ICMP:** معالجة `Type 8 (Echo)` → الرد بـ `Type 0 (Reply)` مع نفس الـ Identifier

#### ✅ معيار النجاح
```
🧪 Testing Network Stack...
  ARP: Resolved 10.0.2.2 → 52:54:00:12:34:56
  ICMP: Received Echo → Sent Reply
  Host ping → Reply received (RTT 2ms)
✅ Network Stack PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| Checksum خاطئ دائماً | راجع ترتيب الـ Bytes (Network Byte Order = Big Endian)، استخدم `htons()` kernel-space |
| الرد لا يصل | تأكد من تحديث `IP ID` و `TTL` بشكل صحيح، وإن الـ MAC Destination هو الـ Host |

---

### 🔹 المهمة 5: الأمان والصلاحيات الأساسي (UID/GID)
#### 📁 الملفات المطلوبة
```
services/
├── security.h / security.c
├── perms.h                (rwx bits, constants)
└── uid_table.c            (جدول العمليات والصلاحيات)
```
#### 🔧 المواصفات التقنية
- **النموذج:** كل عملية لها `uid` و `gid` (الافتراضي `0` للجذر). كل ملف/Inode له `owner_uid`, `group_gid`, `mode` (0755, 0644...)
- **الفحص:** `security_check(inode, operation)` يُستدعى عند `open/read/write/exec`
- **النظام:** `sys_setuid()`, `sys_chmod()`, `sys_chown()`

#### ✅ معيار النجاح
```
🧪 Testing Security Model...
  Root (uid=0) reads secret.txt → OK
  User (uid=1000) reads secret.txt (mode=0600) → Permission Denied
✅ Security Model PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| تجاوز الصلاحيات | تأكد إن الفحص بيحصل في `vfs_open` مش في `read` فقط، وإن الـ bits بتقارن بـ `&` صحيح |
| تعارض UID/GID | أضف validation: `uid < MAX_UID`، وارجع `EINVAL` لو خاطئ |

---

### 🔹 المهمة 6: عملية التهيئة (`init`) وإدارة الخدمات
#### 📁 الملفات المطلوبة
```
services/
├── init.h / init.c        (PID 1 logic)
├── service_mgr.h / service_mgr.c
└── config_parser.c        (قراءة /etc/init.cfg)
```
#### 🔧 المواصفات التقنية
- **PID 1:** أول عملية تنطلق بعد الـ Kernel. مسؤولة عن إطلاق الخدمات الخلفية (Logger, Compat Daemon, Net Manager).
- **المراقبة:** Loop كل 5 ثوانٍ يتفقد حالة الخدمات. لو خدمة وقعت (`EXITED`) يعيد إطلاقها.
- **التكوين:** ملف نصي بسيط `key=value` يحدد الخدمات المفعلة.

#### ✅ معيار النجاح النهائي
```
🚀 Layer 4 Integration Started...
  Init: PID 1 spawned
  Services: [VFS] [FAT32] [IPC] [Net] [Security] → All READY
  Kill Net Service → Init detects crash → Restarts in 2s
  Uptime: 15 min → 0 Panics, 0 Leaks
✅ Layer 4: SYSTEM SERVICES STABLE
🎯 Ready for Layer 5 (EXE Compatibility)
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| خدمة تدخل في Crash Loop | أضف `crash_count` و `backoff_delay` (2s → 4s → 8s) قبل إعادة التشغيل |
| Config file لا يُقرأ | تأكد من تحميل الملف قبل دخول الـ Main Loop، واستخدم `vfs_read` موثوق |

## 🛡️ **البروتوكول الوقائي المدمج في الطبقة 4**

1. ✅ **Minimal Viable Services:** FAT32 قراءة فقط أولاً، IPv4 بدون TCP، IPC Pipes فقط. التوسع لاحقاً.
2. ✅ **Kernel-Mediated:** كل الخدمات هنا Kernel-Space حالياً، لكن الواجهات (`syscalls`) مصممة لتستقبل User-Space لاحقاً.
3. ✅ **Testing Checklist:** كل مهمة فيها `test_*()` تطبع النتيجة على Serial. لا انتقال بدون `PASSED`.
4. ✅ **Clear Boundaries:** لا خلط بين منطق الشبكة وملفات FAT32. كل خدمة معزولة بـ Module Loader بسيط.
5. ✅ **Documentation:** كل ملف يبدأ بـ Header يوضح الطبقة، الوظيفة، والـ dependencies.

## 🚀 **الخطوة التالية**

الطبقة دي هي **العمود الفقري** اللي هيشيل كل اللي جاي. أنصحك بالترتيب ده للتنفيذ الآمن:
1. **VFS (Task 1)** ← الأساس اللي هيربط كل حاجة
2. **FAT32 (Task 2)** ← عشان يكون فيه تخزين حقيقي نختبر عليه
3. **IPC (Task 3)** ← عشان العمليات تتكلم مع بعضها بأمان
4. **Security (Task 5)** ← سهل ويحمي اللي قبله
5. **Net Stack (Task 4)** ← معقد، ييجي بعد استقرار الأساس
6. **Init (Task 6)** ← اللمسة الأخيرة لربط كل الخدمات

**تحب نبدأ فوراً بكود المهمة 1 (VFS Core) مع ملفات `Makefile` محدثة وسكربت اختبار؟** ولا تحب نعدل حاجة في الحدود قبل ما نغوص؟

أنا معاك يا بطل، وهنخلي "مدار" يبني طبقة تلو التانية بثبات. 💪
