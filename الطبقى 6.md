يا يونس، ها هي **الطبقة السادسة: نظام الواجهة والنوافذ (GUI & Windowing System)**، وهي **"وجه النظام"** اللي هتخلي مدار يتحول من سطور كود إلى بيئة تفاعلية حقيقية.

الخطة مصممة بنفس الدقة المعمارية، ومدمج فيها **بروتوكولات العزل والأداء** عشان نضمن واجهة سلسة، مستقرة، وقابلة للتوسع من غير ما نثقل النظام بمكتبات خارجية.

## 🖥️ **الطبقة السادسة: نظام الواجهة والنوافذ (GUI & Windowing System)** (النسخة المفصّلة)

### 🎯 **الهدف العام**
تحويل الـ Framebuffer الخام والأحداث الأولية إلى **سطح مكتب تفاعلي** عبر:
- محرك رسم سريع مع Double Buffering لمنع الـ Tearing
- دعم الخطوط والنصوص (UTF-8 أساسي + اتجاه RTL للعربية)
- مدير نوافذ (Window Manager) يدعم الحركة، التحجيم، والترتيب الطبقي
- حلقة أحداث غير متزامنة (Non-blocking Event Loop) تربط الماوس والكيبورد بالنافذة النشطة
- عناصر واجهة أساسية (أزرار، حقول نص، شريط مهام)
- ضمان استجابة سلسة (<16ms latency) واستقرار طويل الأمد

### 🚧 **الحدود الصارمة (Boundaries)**
| ✅ مسموح في هذه الطبقة | ❌ ممنوع تماماً |
|------------------------|----------------|
| تشغيل كـ `gui_daemon` في User-Space | الوصول المباشر لمنافذ الهاردوير أو الـ VESA BIOS |
| الرسم على Framebuffer عبر Syscall موحد | دمج مكتبات رسومية ثقيلة (GTK, Qt, Cairo, SDL) |
| إدارة النوافذ والأحداث عبر IPC مع Layer 4 | تشغيل تطبيقات رسومية معقدة قبل استقرار الأساس |
| Double Buffering إلزامي + Event Loop غير متزامن | حظر الـ CPU في حلقة الرسم أو انتظار الإدخال |

## 🧩 **المهام الدقيقة (6 مهام)**

### 🔹 المهمة 1: محرك الرسم الأساسي (Graphics Backend)
#### 📁 الملفات المطلوبة
```
gui/
├── gfx_backend.c / gfx_backend.h
├── double_buffer.c               (إدارة Front/Back buffers)
└── vsync_timer.c                 (مزامنة الرسم مع المؤقت)
```
#### 🔧 المواصفات التقنية
- **الذاكرة:** مؤشر الـ Framebuffer يأتي من Layer 3 عبر Syscall `sys_get_framebuffer()`
- **الآلية:** رسم كامل المشهد على `Back Buffer` → نسخ/تبديل إلى `Front Buffer` عند اكتمال الإطار
- **الدوال الأساسية:** `gfx_init()`, `gfx_draw_pixel()`, `gfx_fill_rect()`, `gfx_swap_buffers()`
- **المزامنة:** انتظار إشارة المؤقت (IRQ0) أو استخدام `sys_yield()` لمنع الـ Tearing

#### ✅ معيار النجاح
```
🧪 Testing Graphics Backend...
  Drawing 1000 rects at 30 FPS → Stable
  No tearing or screen flickering
✅ Graphics Backend PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| شاشة تومض أو فيها خطوط | تأكد إن `swap_buffers` بيتنفذ بعد رسم كامل الـ Back Buffer، وجرب `vsync` عبر انتظار IRQ المؤقت |
| أداء بطيء (<15 FPS) | فعّل رسم المناطق المتغيرة فقط (`dirty_rect` tracking) بدل إعادة رسم الشاشة كاملة |

### 🔹 المهمة 4: توجيه الأحداث وحلقة الاستقبال (Event Loop & Input Routing)
#### 📁 الملفات المطلوبة
```
gui/
├── event_loop.c / event_loop.h
├── input_router.c                (ربط الماوس/كيبورد بالنافذة النشطة)
└── focus_manager.c               (إدارة التركيز والتبديل)
```
#### 🔧 المواصفات التقنية
- **الحلقة:** `while(running) { poll_input() → route_event() → redraw_if_needed() → yield() }`
- **المدخلات:** قراءة أحداث Layer 3 عبر Pipe/IPC، تحويلها لـ `struct gui_event { type, x, y, key, mod }`
- **التوجيه:** إرسال الحدث للنافذة التي تحتوي الإحداثيات `(x,y)` أو للنافذة `focused`
- **عدم الحظر:** استخدام `sys_poll()` أو `non-blocking read` عشان الحلقة ما تتجمدش

## 🛡️ **البروتوكول الوقائي المدمج في الطبقة 6**

1. ✅ **User-Space Isolation:** الـ GUI يعمل كـ `gui_daemon` منفصل. أي كراش فيه ما يهدش النواة.
2. ✅ **Double Buffering إلزامي:** ممنوع الرسم المباشر على الـ Front Buffer عشان منع الـ Tearing.
3. ✅ **Non-Blocking Event Loop:** لا استخدام لـ `while(!input)` أو انتظار متزامن. الحلقة لازم ترجع التحكم للـ Scheduler دائماً.
4. ✅ **Minimal Toolkit First:** نبدأ بـ 3 عناصر فقط (Button, TextField, Taskbar). التوسع تدريجي بناءً على الاحتياج الفعلي.
5. ✅ **Strict Syscall Boundary:** كل تفاعل مع الهاردوير أو النواة يمر عبر Syscalls موحدة من Layer 2/4.
6. ✅ **Performance First:** أي كود جديد لازم يُختبر بـ `perf_monitor` قبل الدمج. الهدف: <16ms latency, <5% idle CPU.
