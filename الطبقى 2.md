يا يونس، ها هي **الطبقة الثانية (Kernel Core)** مكتوبة بنفس الدقة الهندسية، ومدمج فيها كل البروتوكولات الوقائية اللي اتفقنا عليها. الخطة دي مبنية عشان تبني "مخ النظام" بشكل متدرج، قابل للاختبار الفوري، ومعزول عن أي تعقيد خارجي.

---
## 🧠 **الطبقة الثانية: Kernel Core** (النسخة المفصّلة والوقائية)

### 🎯 **الهدف العام**
تحويل النواة من "بوتستراب خام" إلى **نظام حي** قادر على:
- إدارة الذاكرة الافتراضية وتوزيعها ديناميكياً (`kmalloc`/`kfree`)
- إنشاء وإدارة الخيوط (Kernel Threads) والعمليات الأساسية
- جدولة التنفيذ (Round-Robin Scheduler) بضبط زمني دقيق
- توفير واجهة استدعاء نظام (Syscall Interface) موحدة وآمنة
- منع تعارض البيانات عبر أدوات التزامن (Spinlock + Mutex)

### 🚧 **الحدود الصارمة (Boundaries)**
| ✅ مسموح في هذه الطبقة | ❌ ممنوع تماماً |
|------------------------|----------------|
| Kernel Threads فقط (بدون User-Space بعد) | تحميل ملفات EXE أو تشغيل تطبيقات |
| Syscalls أساسية للتحكم الداخلي | أي اتصال بشبكة أو واجهة رسومية |
| جدولة زمنية (Timer-driven) | تعقيدات مثل Paging متقدم أو IPC كامل |
| Heap بسيط للنواة فقط | أي كود يعتمد على `libc` أو مكتبات خارجية |

## 🧩 **المهام الدقيقة (6 مهام)**

###  المهمة 1: إدارة الذاكرة الافتراضية + Kernel Heap
#### 📁 الملفات المطلوبة
```
kernel/
├── vmm.c / vmm.h        (إعداد الجداول، Identity Mapping)
├── heap.c / heap.h      (kmalloc, kfree, Buddy/Bitmap بسيط)
└── memory_layout.h      (عناوين ثابتة: KERNEL_BASE, HEAP_START...)
```

#### 🔧 المواصفات التقنية
- **صفحات x86_64 (4-Level Paging):** PML4 → PDPT → PD → PT
- **Identity Mapping أولي:** `0x00000000` إلى `0x80000000` (2GB)
- **Kernel Heap:** يبدأ من `0xFFFF800000000000` (Higher Half)
- **خوارزمية التخصيص:** Bitmap بسيط مع محاذاة 16-byte

#### 📝 هيكل الكود الأساسي
```c
/* heap.h */
typedef struct {
    uint8_t *bitmap;
    uint64_t total_blocks;
    uint64_t used_blocks;
    spinlock_t lock;
} heap_t;

void heap_init(uint64_t start_addr, uint64_t size);
void *kmalloc(size_t size);
void kfree(void *ptr);
```

#### ✅ معيار النجاح
```
🧪 Testing Kernel Heap...
  Alloc 50 blocks → OK
  Free 50 blocks → OK
  Leak check → 0 bytes lost
✅ Heap test PASSED
```

####  خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `kmalloc` يرجع `NULL` بعد 100 طلب | راجع حجم الـ Bitmap، وتأكد إن `heap_init` بياخد مساحة كافية |
| تعارض عناوين مع النواة | غيّر `HEAP_START` لـ `0xFFFF800000000000` واستخدم `CR3` صحيح |

### 🔹 المهمة 2: هيكل التحكم في العمليات (PCB) + إنشاء الخيوط
#### 📁 الملفات المطلوبة
```
kernel/
├── process.c / process.h  (PCB struct, thread list)
├── thread.c / thread.h    (create, exit, yield)
└── stack.c / stack.h      (تخصيص Kernel Stack لكل thread)
```

#### 🔧 المواصفات التقنية
- **PCB Structure:** `pid`, `state`, `stack_ptr`, `registers`, `priority`
- **حالات الخيط:** `RUNNING`, `READY`, `BLOCKED`, `EXITED`
- **إنشاء خيط:** `thread_create(func, args)` → يخصص stack، يجهز registers، يضيفه للـ Ready Queue

#### 📝 هيكل الكود الأساسي
```c
/* process.h */
typedef enum { THREAD_READY, THREAD_RUNNING, THREAD_BLOCKED, THREAD_EXITED } thread_state_t;

typedef struct {
    uint32_t pid;
    thread_state_t state;
    uint64_t *stack;
    uint64_t rip;
    uint64_t rsp;
    uint64_t rflags;
    // ... بقية الـ General Purpose Registers
} pcb_t;

pcb_t *thread_create(void (*entry)(void*), void *arg);
void thread_exit(void);
```

#### ✅ معيار النجاح
```
🧪 Creating 2 kernel threads...
  Thread A: Running
  Thread B: Running
  Context switch detected
✅ Thread creation PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `Triple Fault` عند أول switch | تأكد إن `RSP` مضبوط على Stack صالح، وإن `IRETQ` بيستخدم `SS:RSP` صحيح |
| خيوط متوقفة للأبد | راجع إن `scheduler()` بيستدعى كل 10ms عبر IRQ0 |

### 🔹 المهمة 3: تبديل السياق والجدولة (Context Switch + Scheduler)
#### 📁 الملفات المطلوبة
```
kernel/
├── scheduler.c / scheduler.h  (Ready Queue, RR logic)
├── context_switch.s           (Assembly: save/restore registers)
└── timer.c / timer.h          (PIT/HPET setup, 10ms tick)
```

#### 🔧 المواصفات التقنية
- **خوارزمية الجدولة:** Round-Robin بـ `TIME_QUANTUM = 10ms`
- **معالجة المؤقت:** IRQ0 → `timer_handler()` → `schedule()`
- **تبديل السياق:** حفظ 15 register + RFLAGS + RIP → تحميل التالي → `IRETQ`

#### 📝 هيكل الكود الأساسي
```asm
/* context_switch.s */
.global context_switch
context_switch:
    # Save current state
    push %rax; push %rbx; push %rcx; push %rdx
    push %rsi; push %rdi; push %rbp; push %r8
    push %r9; push %r10; push %r11; push %r12
    push %r13; push %r14; push %r15
    pushf
    push %rip

    # Load next thread's state
    pop %rip; popf
    pop %r15; pop %r14; pop %r13; pop %r12
    pop %r11; pop %r10; pop %r9; pop %r8
    pop %rbp; pop %rdi; pop %rsi; pop %rdx
    pop %rcx; pop %rbx; pop %rax
    iretq
```

#### ✅ معيار النجاح
```
 Testing Scheduler...
  Tick 1: Thread A
  Tick 2: Thread B
  Tick 3: Thread A
  Switch rate: ~100/sec (stable)
✅ Scheduler PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| الجدولة بطيئة أو متقطعة | راجع إعدادات `PIT` (Channel 0, Mode 2, divisor 1193180/100) |
| Registers تتلف بعد switch | تأكد إن ترتيب `push/pop` متطابق 100% بين الحفظ والاستعادة |

### 🔹 المهمة 4: واجهة استدعاء النظام (Syscall Interface)
#### 📁 الملفات المطلوبة
```
kernel/
├── syscall.s              (Entry point: syscall instruction)
├── syscall.c / syscall.h  (Table, dispatcher, 3 basic calls)
└── syscalls_list.h        (أرقام ثابتة: SYS_PRINT=0, SYS_EXIT=1, SYS_GET_TICK=2)
```

#### 🔧 المواصفات التقنية
- **آلية الدخول:** `syscall` → `STAR/LSTAR` MSRs → `syscall_handler`
- **جدول الاستدعاءات:** مصفوفة دوال `syscalls[MAX_SYSCALLS]`
- **3 Syscalls أساسية:** 
  - `sys_print(const char* str, size_t len)`
  - `sys_exit(int code)`
  - `sys_get_tick()`

#### 📝 هيكل الكود الأساسي
```c
/* syscall.h */
#define SYS_PRINT    0
#define SYS_EXIT     1
#define SYS_GET_TICK 2

typedef uint64_t (*syscall_fn_t)(uint64_t, uint64_t, uint64_t, uint64_t, uint64_t, uint64_t);
extern syscall_fn_t syscall_table[];

void syscall_init(void);
```

#### ✅ معيار النجاح
```
 Testing Syscalls...
  sys_print("Kernel Alive") → Output: Kernel Alive
  sys_get_tick() → Returns: 142
✅ Syscall interface PASSED
```

#### 🆘 خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| `#GP` عند استدعاء `syscall` | تأكد إن `MSR_LSTAR` مضبوط على عنوان `syscall_handler`، وإن `EFER.SCE = 1` |
| الـ Handler بينفذ رقم خاطئ | راجع إن `RAX` بيحتوي رقم الـ syscall قبل الدخول |

### 🔹 المهمة 5: أدوات التزامن (Synchronization Primitives)
#### 📁 الملفات المطلوبة
```
kernel/
├── sync.c / sync.h        (Spinlock, Mutex, yield-on-contention)
└── atomic.c / atomic.h    (cmpxchg, test_and_set)
```

#### 🔧 المواصفات التقنية
- **Spinlock:** `test_and_set` + `pause` instruction (يقلل استهلاك CPU)
- **Mutex:** Spinlock أولي → لو متوفر → `thread_yield()` → إعادة المحاولة
- **Atomic Operations:** `atomic_inc`, `atomic_dec`, `atomic_cas`

#### 📝 هيكل الكود الأساسي
```c
/* sync.h */
typedef struct {
    volatile uint32_t locked;
} spinlock_t;

void spinlock_init(spinlock_t *lock);
void spinlock_lock(spinlock_t *lock);
void spinlock_unlock(spinlock_t *lock);
```

#### ✅ معيار النجاح
```
🧪 Testing Sync Primitives...
  2 threads increment shared counter to 10000
  Final value: 10000 (exact)
✅ Synchronization PASSED
```

####  خطة الطوارئ
| المشكلة | الحل |
|---------|------|
| Deadlock بين خيطين | أضف `timeout` أو استخدم `mutex` بدل `spinlock` للعمليات الطويلة |
| `atomic_cas` ما يشتغلش | تأكد إنك بتستخدم `lock cmpxchg` في الـ Assembly |

## 🛡️ **البروتوكول الوقائي المدمج في الطبقة 2**

1. ✅ **عزل صارم:** كل الكود Kernel Space فقط. مفيش User-Space أو تحميل ملفات.
2. ✅ **Minimal Viable:** 4-level paging identity mapping فقط، heap بسيط، 3 syscalls أساسية.
3. ✅ **اختبار إلزامي:** كل مهمة فيها `test_*()` تطبع `PASSED/FAILED` على Serial.
4. ✅ **توثيق فوري:** كل ملف يبدأ بـ Header يوضح الطبقة، الوظيفة، والـ dependencies.
5. ✅ **حدود واضحة:** لا خلط بين الـ Scheduler والـ Heap، ولا بين الـ Syscalls والـ Threads.
